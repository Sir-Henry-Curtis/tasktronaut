Tasktronaut is a controlled VS Code coding assistant for internal engineering workflows. You describe what you want in plain text, and Tasktronaut can read files, edit files, run terminal commands, fetch approved URLs, and use configured MCP tools while preserving human approval gates.

Think of it as pair programming with an AI that can touch your workspace and run your tools, but only inside the deployment boundary you configure.

This guide gets a developer from a packaged Tasktronaut build to a first approved task.

<Info>
  **What You'll Need:**
  - VS Code or a compatible VS Code-based editor
  - A Tasktronaut VSIX or approved internal extension source
  - An approved model endpoint and credential, or an approved local model host
  - A workspace under version control
</Info>

## 1. Install

Install the approved Tasktronaut package using your organization's normal extension distribution path.

For a local VSIX, open the Extensions view, choose **Install from VSIX...**, and select the packaged Tasktronaut build. Reload VS Code when prompted.

## 2. Configure the Model Endpoint

Open the Tasktronaut panel and go to settings. Configure the approved provider for each mode you plan to use:

- **Plan** for investigation and strategy
- **Act** for tool-using implementation
- **KISS** for lightweight conversational local models

Tasktronaut's baseline internal deployment does not require a hosted Cline account. Credentials should point at the endpoint approved for your environment.

<Note>
Use the provider and model settings supplied by your administrator or project lead. Avoid adding personal cloud providers unless they are approved for the workspace.
</Note>

## 3. Choose a Mode

Use the mode control at the bottom of the chat input:

- **Plan** when you want Tasktronaut to inspect, explain, or propose a strategy before changing files
- **Act** when you are ready for Tasktronaut to edit files, run commands, or use tools
- **KISS** when you want a minimal conversational prompt for a small local model

## 4. Run a First Task

Open a version-controlled workspace, then paste:

```text
Create a Python function that checks if a string is a palindrome. Include tests.
```

In Act mode, Tasktronaut analyzes the request, proposes file changes, and asks for approval before writing. Approve the edits you want, reject anything that does not fit, and let it run tests only when the command is appropriate for the workspace.

That's it. You have working code with tests.

<Note>
**Not seeing file changes?** You may be in Plan mode or KISS mode. Switch to Act when you want Tasktronaut to modify files.
</Note>

## 5. You're All Set

Let's recap what you just did:

1. Installed Tasktronaut in your editor
2. Configured an approved model endpoint
3. Chose the right mode for the task
4. Built working code with approval gates

Every step of the way, Tasktronaut shows what it plans to do and waits for approval unless you explicitly enable broader auto-approval. You can review file edits, terminal commands, URL fetches, and MCP tool calls before they run.

## What Else Can Tasktronaut Do?

The palindrome example is just the beginning. Try:

- `Read my project and explain the architecture`
- `Find and fix the bug in the parser module`
- `Run my test suite and summarize any failures`
- `Fetch the official docs URL I pasted and compare it with this implementation`
- `Start a GSD planning workflow for this feature`

Tasktronaut works best when you give it clear, specific instructions and keep the workspace under version control.

<AccordionGroup>
  <Accordion title="How does Tasktronaut work under the hood?">
    Tasktronaut runs in your editor. When you send a message, it goes to the configured model endpoint. The model response is parsed into actions such as file edits, terminal commands, controlled URL fetches, or MCP tool calls. Tasktronaut presents consequential actions for approval before running them.
  </Accordion>
</AccordionGroup>
